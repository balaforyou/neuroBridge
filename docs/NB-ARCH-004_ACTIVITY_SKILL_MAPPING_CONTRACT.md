# NB-ARCH-004 Activity Skill Mapping Contract

Status: Done

## 1. Purpose

This document defines the NeuroBridge-wide contract for how activities publish cognitive skill evidence for analytics and parent-facing interpretation.

The contract prevents static one-size-fits-all activity skill mapping. Activities should publish skill evidence from the actual learner experience, including the activity, mode, level, and scaffold or progression stage that produced the evidence.

## 2. Problem Statement

The existing architecture supports activity metadata, skills, domains, and analytics aggregation. A static activity-to-skill list is useful for discovery, but it is not precise enough for insight-rich analytics.

The same activity can produce different cognitive evidence depending on how the learner experiences it. For example, Schulte Table in ascending mode provides different evidence than descending mode, memory mode, or Listen & Find mode.

If analytics aggregate only by activity score, future consumers may overgeneralize learner performance. A strong activity score does not always mean the same skill improved, and a difficult mode may reveal a different support need than the base activity.

## 3. Core Principle

Skills are not fixed properties of an activity.

Skills are evidence produced by a specific learner experience:

```text
activity + mode + level + scaffold/progression stage
```

Activity metadata should therefore describe the skill evidence generated by a specific experience slice, not only the broad activity family.

## 4. Mapping Model

NeuroBridge activities should publish skill evidence at the smallest meaningful experience unit.

Recommended mapping path:

```text
Activity Family
  -> Activity
  -> Mode
  -> Level
  -> Progression Stage
  -> Scaffold State
  -> Skill Evidence
```

Skill evidence can then be aggregated by analytics consumers across sessions, activities, modes, and time windows.

The model supports:

- Different skills for different modes in the same activity.
- Different evidence weights for introductory, scaffolded, practice, memory, mastery, or fluency experiences.
- Parent-facing summaries that explain what the learner practiced, not only which activity was played.
- Future recommendations that select the next activity based on skill evidence gaps.

## 5. Required Fields

Activity metadata should support the following fields when publishing skill evidence:

- `activityId`
- `activityFamily`
- `activityName`
- `mode`
- `level`
- `progressionStage`
- `primaryDomain`
- `primarySkill`
- `subSkills`
- `evidenceWeight`
- `scaffoldState`
- `analyticsTags`

Field intent:

- `activityId`: Stable internal activity identifier.
- `activityFamily`: Broader activity family, such as Schulte Family or Matching Worksheets.
- `activityName`: Parent or learner-facing activity name as appropriate for the analytics surface.
- `mode`: Specific learner experience mode, such as Ascending, Descending, Memory Mode, or Listen & Find.
- `level`: Current difficulty level, board size, stage, or equivalent activity progression unit.
- `progressionStage`: Shared NeuroBridge progression stage such as Learn, Practice, Memory, Mastery, or Fluency.
- `primaryDomain`: Broad cognitive or learning domain supported by this experience slice.
- `primarySkill`: Main skill evidenced by this experience slice.
- `subSkills`: Additional skills evidenced by the same experience slice.
- `evidenceWeight`: Relative strength of evidence produced by this experience slice.
- `scaffoldState`: Current support level, hint state, or scaffold fading state.
- `analyticsTags`: Additional structured tags for aggregation, filtering, and future interpretation.

## 6. Schulte Example

Schulte Table should not publish only one static skill mapping. Skill evidence should vary by mode and progression stage.

Example evidence:

| Mode | Example Skill Evidence |
| --- | --- |
| Ascending | Visual Search, Visual Scanning, Selective Attention |
| Descending | Cognitive Flexibility, Reverse Sequencing, Working Memory |
| Memory Mode | Working Memory, Mental Tracking, Sequential Recall |
| Listen & Find | Auditory Processing, Cross-modal Integration, Visual Search |

Example metadata shape:

```json
{
  "activityId": "schulte",
  "activityFamily": "Schulte Family",
  "activityName": "Schulte Table",
  "mode": "Listen & Find",
  "level": "3x3",
  "progressionStage": "Practice",
  "primaryDomain": "attention",
  "primarySkill": "auditory-processing",
  "subSkills": [
    "cross-modal-integration",
    "visual-search"
  ],
  "evidenceWeight": "medium",
  "scaffoldState": "visual-prompt-with-audio",
  "analyticsTags": [
    "ordered-prompt",
    "number-search",
    "audio-supported"
  ]
}
```

This example is illustrative. It does not retrofit current Schulte analytics or freeze the final cognitive taxonomy.

## 7. Future Activity Examples

Future activities should follow the same experience-slice model.

Examples:

- Attribute Explorer: evidence may vary between same/different comparison, attribute isolation, rule discovery, and scaffolded hint states.
- Pattern Explorer: evidence may vary by pattern type, sequence complexity, missing-item position, and progression stage.
- Who What Where: evidence may vary by language demand, scene complexity, role identification, and working memory load.
- Sequencing: evidence may vary by number of steps, visual support, temporal reasoning demand, and correction path.
- Matching Worksheets: evidence may vary by attribute category, distractor similarity, correction review, and independent versus scaffolded completion.
- Number Bridges: evidence may vary by operation, level, strategy scaffold, fluency pressure, and mastery status.

## 8. Analytics Consumers

Future analytics should aggregate by skill evidence, not only by activity score.

Primary consumers include:

- Parent Dashboard
- Cognitive Snapshot
- Learning Signals
- Adaptive Progression
- SIRAASH future recommendations

Consumer expectations:

- Parent Dashboard should help parents understand what skills were practiced and where support may be needed.
- Cognitive Snapshot should summarize evidence across activities, modes, levels, and scaffold states.
- Learning Signals should detect improving, stable, difficult, or neglected skill areas.
- Adaptive Progression should use skill evidence before changing difficulty or recommending mastery.
- SIRAASH future recommendations should suggest activities based on observed evidence gaps and recent learner experience.

## 9. Out of Scope

This contract does not implement runtime behavior.

Out of scope:

- Code changes.
- Retrofitting Schulte analytics now.
- Finalizing the full cognitive taxonomy.
- Parent dashboard changes.
- Analytics schema migration.
- Activity-specific implementation changes.
- Changing current activity scores or completion summaries.

## 10. Implementation Guidance

Future implementation should use this contract as an architecture guide when extending activity metadata, analytics payloads, or parent-facing summaries.

Guidance:

- Keep stable activity IDs and route names unchanged unless a separate architecture decision approves a change.
- Add skill evidence metadata at the activity, mode, level, and progression-stage boundary where it naturally belongs.
- Keep learner-facing screens calm and simple; publish deeper interpretation to parent and analytics surfaces.
- Treat scaffold state as part of the evidence context, because independent and scaffolded performance do not carry the same meaning.
- Prefer structured metadata over inferring cognitive meaning from raw score, timing, or trial shape.
- Use evidence weights to avoid overstating weak signals from early, heavily scaffolded, or very short experiences.
- Create follow-up backlog items for taxonomy gaps instead of forcing premature skill names into implementation packets.
- Preserve the existing analytics core until a specific implementation packet authorizes schema or payload changes.
